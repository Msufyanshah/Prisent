# PHR — Progress Health Reports

> Version: 2.0 | Layer 08 of SpecKit-Plus
> A PHR is filed after every completed task — not at end of day.
> PHR status FAIL or PARTIAL = next task is blocked until resolved.

---

## Why PHR Exists

Traditional development testing at the end creates:
- Bug accumulation across multiple tasks
- Hallucinated implementations that "look right" but don't work
- Frontend-backend integration failures discovered too late
- Wasted time unwinding 5 tasks to fix 1 root cause

PHR forces verification at the smallest possible unit.
One task. One test. One report. Move forward only when it passes.

---

## PHR Template

Copy this for every task. File as `PHR-[TASK-ID].md`.

```markdown
# PHR — TASK-XXX [Task Title]

**Date:** YYYY-MM-DD
**Day:** D[N]
**Time started:** HH:MM
**Time completed:** HH:MM
**Time taken:** Xh Xmin
**Estimated time:** Xh
**Variance:** +/- Xh (over/under)

---

## Smoke Test Results

| Test | Command | Expected | Actual | Result |
|---|---|---|---|---|
| 1 | `command here` | expected output | actual output | ✅ PASS / ❌ FAIL |
| 2 | `command here` | expected output | actual output | ✅ PASS / ❌ FAIL |
| 3 | `command here` | expected output | actual output | ✅ PASS / ❌ FAIL |

---

## Overall Status

**Status:** ✅ PASS | ⚠️ PARTIAL | ❌ FAIL

---

## Deviations from Spec

> Did the implementation deviate from the Contracts or Specification?

- [ ] No deviations — implemented exactly as specified
- [ ] Minor deviation: [describe]
- [ ] Major deviation: [describe — must update Contracts/Spec]

---

## Blockers for Next Task

> Is anything blocking the next task?

- [ ] No blockers — next task can start immediately
- [ ] Blocker: [describe the blocker and plan to resolve]

---

## Notes

> Anything unusual, surprising, or worth remembering.

[Free text]

---

## Action Required

> What must happen before next task starts?

- [ ] None — PASS, proceed to TASK-XXX
- [ ] Fix failing smoke test [X] — then re-run before proceeding
- [ ] Update CONTRACTS.md — deviation documented
- [ ] Add ADR to DECISIONS.md — tech choice made
- [ ] Escalate blocker — stuck more than 30 min on: [describe]
```

### PHR-013 (TASK-013 — Research Agent) — Example of a FAIL

```markdown
# PHR — TASK-013 Research Agent

Date: June 2026
Day: D3
Time taken: 3h 30min
Estimated: 2.5h
Variance: +1h (OVER)

## Smoke Test Results
| Test | Expected | Actual | Result |
|---|---|---|---|
| 1: status == success | "success" | "success" | ✅ PASS |
| 2: content_pillar valid | matches user pillars | Got "General" | ❌ FAIL |
| 3: agent under 30s | < 30s | 47 seconds | ❌ FAIL |

Status: ❌ FAIL
Deviations:
  - content_pillar validation not working — agent ignores instruction
  - Response time too slow — GPT-4o-mini doing 5 tool calls unnecessarily

Blockers: TASK-014 (Writer Agent) depends on Research output being valid
Root cause:
  1. Pillar validation: System prompt not explicit enough — need to add examples
  2. Speed: Mock web_search returning too much text — agent over-searches

Fix plan:
  1. Add explicit pillar examples to system prompt
  2. Truncate mock_web_search output to 200 chars
  3. Add max_tool_calls=3 to agent loop
  4. Re-run smoke test

Action: Fix both issues, re-run all smoke tests. Do NOT start TASK-014 until PASS.
```

## Daily Integration PHR

At end of every day, file one additional PHR: `PHR-D[N]-INTEGRATION.md`

```markdown
# PHR — Day [N] Integration Checkpoint

**Tasks completed today:** TASK-XXX, TASK-XXX, ...
**All individual PHRs filed:** Yes / No

## End-to-End Integration Test
[paste the integration checkpoint command from TASKS.md and its output]

## Integration Status: ✅ PASS | ❌ FAIL

## Carry-forward issues (if any)
[Any PARTIAL tasks or unresolved blockers going into tomorrow]

## Tomorrow's readiness
- [ ] All today's smoke tests passing
- [ ] No unresolved FAIL PHRs
- [ ] Know exactly which task to start tomorrow
- [ ] Contracts/Spec updated if any deviations logged
```
